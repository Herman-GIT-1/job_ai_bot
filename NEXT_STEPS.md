# Рекомендации по развитию проекта

Текущее состояние: MVP работает локально для одного пользователя.
Ниже — конкретные шаги от текущего состояния до готового продукта.

---

## Приоритет 1 — Качество работы (сейчас)

### 1.1 Добавить описание вакансии в промпты
Сейчас `ai_score.py` и `cover_letter.py` видят только `title`, `company`, `tech_stack`.
SerpAPI возвращает полное `description` — передавай его в промпт. Оценка станет точнее,
cover letter — конкретнее.

```python
# scraper.py — сохранять description
jobs.append({..., "description": job.get("description", "")[:1500]})

# database.py — добавить колонку
ALTER TABLE jobs ADD COLUMN description TEXT;
```

### 1.2 Кэшировать резюме в памяти
`load_resume()` читает файл с диска при каждом вызове `evaluate()` и `generate_letter()`.
При `/score` на 50 вакансий — 100 лишних I/O операций. Достаточно читать один раз в начале
`run_score()` / `_run()` и передавать как параметр.

### 1.3 Обработка ошибок Groq в `/scrape`
Если `build_queries()` вернул fallback-запросы из-за ошибки Groq — сообщить пользователю
в Telegram, чтобы он понимал почему результаты могут быть менее точными.

## Приоритет 2 — Переход на Stage 2 (первые внешние пользователи)

### 2.1 Заменить Groq → Claude API
`llama-3.1-8b-instant` — компромисс ради бесплатного тира. Качество оценки и cover letter
значительно вырастет с `claude-haiku-4-5` (дёшево) или `claude-sonnet-4-6` (точнее).

Контракт модулей уже стабилен — замена сводится к смене клиента и модели в
`ai_score.py`, `cover_letter.py`, `scraper.py`.

### 2.2 PostgreSQL вместо SQLite
При переходе к нескольким пользователям добавить `chat_id` во все таблицы:

```sql
CREATE TABLE users (
    chat_id BIGINT PRIMARY KEY,
    username TEXT,
    resume_text TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

ALTER TABLE jobs ADD COLUMN chat_id BIGINT REFERENCES users(chat_id);
```

`database.py` уже изолирует интерфейс — менять нужно только внутренности модуля.

### 2.3 Хранить резюме в БД, а не в файле
`resume.txt` — временное решение. При нескольких пользователях каждый должен иметь
своё резюме в таблице `users.resume_text`. `resume_parser.py` уже имеет `save_resume()`
и `load_resume()` — достаточно перенаправить их в БД.

### 2.4 Деплой на сервер
Бот должен работать 24/7. Минимальный путь:
- **Railway.app** — бесплатный тир, деплой через GitHub push, переменные окружения через UI
- Альтернатива: VPS (Hetzner ~4€/мес) + `systemd` сервис или Docker

## Приоритет 3 — Улучшение пользовательского опыта

### 3.1 Онбординг через `/start`
Новый пользователь сейчас получает список команд. Лучше вести его за руку:

```
/start → "Привет! Для начала загрузи резюме."
<файл> → "Резюме принято. Теперь введи /scrape и выбери город."
/scrape → ...
```

### 3.2 Фильтр по минимальному score в `/jobs`
Сейчас порог захардкожен как `>= 7` в SQL. Полезно дать пользователю выбор:
`/jobs 6` — показать с оценкой ≥ 6.

### 3.3 Повторная оценка отдельной вакансии
Если резюме обновилось — старые оценки устарели. Добавить кнопку "Переоценить 🔄"
или команду `/rescore` которая сбрасывает `score = NULL` и запускает оценку заново.

### 3.4 Экспорт в CSV
`/export` — выгрузить все вакансии (или только поданные) как CSV файл.
Удобно для отслеживания статуса заявок вне бота.

## Приоритет 4 — Надёжность

### 4.1 Логирование
Сейчас ошибки уходят в `print()`. При деплое на сервер они теряются.
Заменить на `logging` с записью в файл или отправкой в Telegram при критических ошибках.

### 4.2 Rate limiting для SerpAPI
100 запросов/месяц заканчиваются незаметно. Хранить счётчик использованных запросов
в БД и предупреждать пользователя когда остаток < 20.

### 4.3 Ограничение доступа к боту
Сейчас любой кто узнает токен может пользоваться ботом. Добавить проверку `chat_id`
в каждый хендлер пока бот однопользовательский:

```python
if update.effective_chat.id != CHAT_ID:
    await update.message.reply_text("Доступ запрещён.")
    return
```

## Технический долг

| Проблема | Где | Сложность |
|---|---|---|
| `os` импортирован но не используется в `bot.py` | bot.py:1 | Минуты |
| Глобальное соединение SQLite с `check_same_thread=False` — не thread-safe при параллельных записях | database.py | Средняя — нужен `threading.Lock` или connection-per-call |
| `scraper.py` сохраняет `city="Warsaw"` хардкодом в dict, игнорируя переданный `city` параметр | scraper.py:95 | Минуты |
| Нет валидации ответа Groq в `build_queries` — если модель вернёт не-JSON, `json.loads` бросит исключение которое поймает `except Exception` и уйдёт в fallback без явного сообщения | scraper.py | Низкая |
